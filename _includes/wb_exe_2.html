Exercise 2: Documenting Multiple Webhook Events

Scenario: You are documenting webhooks for a **subscription-based SaaS platform** that sends notifications for various billing events.

Your Task: Document the following webhook events:

  • **subscription.created** – Triggered when a user subscribes.
  • **subscription.renewed** – Fired when a subscription renews automatically.
  • **subscription.canceled** – Sent when a user cancels their subscription.

Webhook Event Details:

Endpoint: POST /webhooks/subscription

Destination: User-provided webhook URL

Example Webhook Payload:
        {
            "event": "subscription.renewed",
            "data": {
                "subscription_id": "SUB78910",
                "user": {
                    "id": "USR12345",
                    "email": "john@example.com"
                },
                "plan": {
                    "name": "Premium",
                    "amount": 49.99,
                    "currency": "USD"
                },
                "status": "active",
                "renewal_date": "2024-03-01T12:00:00Z"
            }
        }

Solution to Exercise 2

1. Overview

The **Subscription Webhooks** notify SaaS applications when users **subscribe, renew, or cancel their subscriptions**. These webhooks help automate billing, analytics, and customer notifications.

2. Webhook Registration

Developers can **subscribe their webhook URLs** using the following API endpoint:

        POST /webhooks/register
        {
            "event": ["subscription.created", "subscription.renewed", "subscription.canceled"],
            "url": "https://yourwebhookurl.com/subscription"
        }

Upon successful registration, they receive a confirmation response.

3. Webhook Events

Event Name Trigger Condition Description
subscription.created When a user subscribes Triggers when a new subscription is created.
subscription.renewed When a subscription renews Fires when a user’s plan renews automatically.
subscription.canceled When a user cancels Notifies when a user stops their subscription.

4. Request Format

When a webhook is triggered, the following payload is sent:

        {
            "event": "subscription.renewed",
            "data": {
                "subscription_id": "SUB78910",
                "user": {
                    "id": "USR12345",
                    "email": "john@example.com"
                },
                "plan": {
                    "name": "Premium",
                    "amount": 49.99,
                    "currency": "USD"
                },
                "status": "active",
                "renewal_date": "2024-03-01T12:00:00Z"
            }
        }

5. Expected Response from Client

The client must return **200 OK** to acknowledge the webhook.

6. Webhook Security - Verifying Webhooks

Webhooks include an **X-Signature** header for security verification.

        import hmac
        import hashlib

        secret = "your-webhook-secret"
        payload = '{"event":"subscription.renewed","data":{"subscription_id":"SUB78910"}}'
        received_signature = "bde67cfaf998c...5af32c"

        calculated_signature = hmac.new(
            key=secret.encode(),
            msg=payload.encode(),
            digestmod=hashlib.sha256
        ).hexdigest()

        if calculated_signature == received_signature:
            print("Webhook verified!")
        else:
            print("Invalid webhook request.")

7. Webhook Retry Policy

If the webhook fails (does not return **200 OK**), we **retry up to 3 times**.

8. Testing Webhooks

Developers can use **Postman** or **cURL** to test webhook responses:

        curl -X POST "https://yourwebhookurl.com/subscription" -H "Content-Type: application/json" -d '{
            "event": "subscription.renewed",
            "data": {
                "subscription_id": "SUB78910",
                "user": {"id": "USR12345", "email": "john@example.com"},

                "status": "active",
                "renewal_date": "2024-03-01T12:00:00Z"
            }
        }'
